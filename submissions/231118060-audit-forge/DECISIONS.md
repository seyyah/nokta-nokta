# Decisions

## Audio Stack

I used `expo-av` because the baseline is already an Expo app and the requirement is live microphone level, not speech transcription. Expo AV exposes recording permission, a native recording session and `metering` in `RecordingStatus`. That gives a real dB signal without adding a speech SDK or server.

I did not generate a fake waveform. The bars read the same smoothed amplitude that drives the avatar mouth. The code now sets `isMeteringEnabled: true` explicitly on the recording options and reports startup errors in the panel, so a missing recorder path is visible instead of being mistaken for silence.

## Lipsync Approach

I used amplitude-driven lipsync. Visemes would require speech recognition or phoneme timing, which would be heavier than the assignment needs and fragile in a small Expo submission. The current approach is honest: it maps live speech energy to mouth openness and keeps silence idle.

## Avatar Render

The app uses a real local GLB asset at `app/assets/avatar.glb`. The asset is generated by `app/scripts/create-avatar-asset.mjs` so it is reproducible and owned by this submission. It contains a named `Mouth` node, which makes the runtime mouth response explicit instead of relying on a generic mesh.

For rendering, I chose `expo-gl` + `three` + `GLTFLoader` with `expo-three`'s renderer wrapper. This keeps the 3D surface contained inside one component and avoids building a large scene framework around one avatar.

## Bridge Tool

I chose Jitsi Meet because it supports audio, video and screen share through a stable external room URL. A native call UI would either need heavier native setup or risk becoming fake. The app launches a real meeting target and documents that it is external.

The bridge button checks whether the URL can be opened and shows the failure inline when the OS/browser handler is unavailable.

## State And File Organization

Feature code stays under `app/src`:

- `audio/` for microphone state
- `components/` for UI and avatar rendering
- `bridge/` for provider configuration
- `types/` for asset declarations

This keeps the old `NoktaScreen` pattern intact and avoids repo-wide configuration changes.

## What I Did Not Do

- I did not move `AuditWidget` into a screen component.
- I did not add a fake call screen.
- I did not claim phoneme-level lipsync.
- I did not touch other submissions or root workflows.
- I did not replace the whole UI with a new dashboard.

## Risk Accepted

The Three/GL path should be verified on a physical Android device. TypeScript can confirm the code shape, but native GL behavior still needs device testing.
